# Confidence Interval Audit — 2026-07-17

## Purpose

Audit of `data/wisspar_export_production.csv` (6,546 data rows at time of audit) to find
rows where `outcome_overview_lower_limit` / `outcome_overview_upper_limit` are inconsistent
with `outcome_overview_value`, and to check for other extreme values suggestive of
transcription or extraction errors. All source verification was done against the
ClinicalTrials.gov API v2 (`resultsSection`), per the project's data extraction rules.

## Method

1. Programmatic scan of the CSV for:
   - `lower_limit > upper_limit` (reversed bounds) — **0 found**.
   - `value` outside `[lower_limit, upper_limit]` — **1,345 found**, of which **1,342** have
     `lower_limit = 0` and `upper_limit = 0` (see "Missing-CI convention" below) and **15**
     have genuinely inconsistent, non-placeholder bounds.
   - Only one of `lower_limit`/`upper_limit` populated — **0 found**.
   - Upper/lower ratio > 50x — **12 found**.
   - Negative values — **0 found**.
   - Values matching a "leading zero, no decimal point" pattern (e.g. `082`) — **1 found**.
2. For every flagged row, fetched `https://clinicaltrials.gov/api/v2/studies/{NCT_ID}?fields=resultsSection`
   and compared the CSV row against the matching outcome measure / class / group measurement
   in the raw API response.

## Confirmed errors and fixes applied

38 rows across 6 trials had values and/or bounds that didn't match the ClinicalTrials.gov
source. All were corrected in place in `data/wisspar_export_production.csv`. The full
before/after values are recorded in `data/ci_audit_corrections_2026_07_17.csv`.

### Bug A — "7 additional serotype" OPA table, class-index shift off by 5
**Trials:** NCT03313037 (VAX-24), NCT04875533 (VAX-31)

These trials report OPA GMTs for 7 non-common serotypes (8, 10A, 11A, 12F, 15B, 22F, 33F) in
a single outcome measure with 7 classes. In both trials, the last two classes (22F, 33F —
indices 5 and 6) had their `lowerLimit`/`upperLimit` populated from classes 0 and 1
(serotype 8, serotype 10A) instead of their own — a consistent "shift by −5" indexing error,
almost certainly from the original manual/legacy extraction rather than the current
`dual_extract.py` pipeline (no `data/extractions/{nct_id}/` artifacts exist for either trial).

- **NCT03313037**: point *values* were also wrong for all 14 rows (didn't match source at
  all) — full re-extraction of value + bounds for both groups (20vPnC, 13vPnC/PPSV23) ×
  7 serotypes.
- **NCT04875533**: point values were correct; only bounds needed fixing — 12 rows had
  `0/0` placeholder bounds populated, 2 rows (22F, 33F for the 20vPnC group) had bounds
  swapped from serotype 8 / 10A.

### Bug B — adjacent-class bound bleed in paired GMC tables
**Trials:** NCT00373958, NCT00366899 (Prevnar13 pivotal infant/toddler trials)

In GMC tables listing "common serotypes" then "additional serotypes" as sequential classes,
serotype 5's row picked up serotype 4's confidence bounds (and, in NCT00366899, serotype 4's
row picked up serotype 1's bounds). Point values were correct in all cases; only bounds were
swapped between adjacent/nearby classes.

### Bug C — post-vaccination value paired with pre-vaccination bounds
**Trials:** NCT00345358 (Synflorix), NCT01616459

In paired pre/post-booster outcome tables (classes alternate PRE, POST, PRE, POST, ...), the
post-boost row's point value was extracted correctly but its CI bounds were copied from the
immediately preceding pre-boost row instead of its own. Affected serotype 6A, both GMC and
OPA, across multiple age subgroups.

### Source data typo (not an extraction bug)
**Trial:** NCT00345358, row for serotype 19A, ≥24M Synflorix group, GMC

ClinicalTrials.gov's own posted results table lists `upperLimit = "082"` (value 0.65, lower
0.53). This is almost certainly a dropped decimal point for `0.82` — an upper bound of 82
would imply a ~126x interval on a GMC of 0.65, which is inconsistent with every comparable
row in the same table, whereas 0.82 fits the pattern exactly. Corrected to `0.82` in the CSV
per user decision; the ClinicalTrials.gov source itself still shows the malformed value.

## Checked and found to be genuine (no fix needed)

**NCT04546425** (Pfizer 20vPnC 3-dose infant study, Russian cohort) — 11 of the 12 rows
flagged for >50x upper/lower ratio. These are OPA GMTs in a small subgroup (n=22–24 infants).
Verified exactly against source: values and bounds match the API response precisely. Wide
CIs here reflect genuine sampling variability in a small subgroup, not a data error.

**Duplicate-value heuristic:** Checked for exact-duplicate point estimates across different
serotypes within the same trial/group/outcome, as a heuristic for the Bug-A-style pattern
found elsewhere. This produced many false positives — pediatric pre-boost GMCs frequently
cluster at the assay's lower limit of quantification (e.g., 0.03–0.05 μg/mL) across multiple
serotypes legitimately, so shared values are expected and not evidence of an error. Abandoned
as a detection signal; no further trials were flagged this way.

## Update — 2026-07-20: "Missing-CI-as-zero" convention was wrong

The 2026-07-17 pass above assumed the 1,342 `lower_limit = 0, upper_limit = 0` rows meant "no
CI reported in source." Re-verification against the live ClinicalTrials.gov API v2
`resultsSection` for every affected trial disproved this: **in every row that could be matched
to source, a real CI was actually reported** — the 0/0 values are dropped-CI extraction
failures, not a deliberate "not reported" placeholder. Zero rows were confirmed as genuinely
CI-less in source.

Of 1,320 current 0/0 rows (1,342 minus 22 already fixed in the 2026-07-17 pass):
- **324 rows** (CSV lines 4118–4441) sit in a block where every trial-identifying field
  (name, NCT ID, sponsor, phase, country) is blank — not a fill-down convention, genuinely
  missing metadata. These could not be attributed to any trial and were excluded from this
  pass. **This is a separate, unresolved data-integrity problem** — see below.
- **751 rows** were matched to source with an unambiguous or value-consistent CI and were
  backfilled with the real bounds. See `data/ci_backfill_2026_07_20.csv` for the full
  line-by-line log (old 0/0, new bounds, matched source class/group, match confidence).
- **104 rows** belong to NCT02736240 (Walvax 13-valent PCV), which currently returns
  `hasResults: false` from the API. Resolved via manuscript — see below.
- **96 rows** could not be matched to any source measurement, most notably 80 OPA rows for
  NCT03512288 — that trial's current API results contain only IgG/GMC outcome measures, no
  OPA table at all. Resolved via manuscript for those 80 rows — see below. Remaining 16 rows
  (9 in NCT00366899, 6 in NCT00373958, 1 in NCT04384107) are still unmatched; at least one
  (NCT00373958, serotype 1, OPA) has a point **value** that doesn't match source at all, not
  just the bounds — a deeper extraction failure than a dropped CI. Not fixed; needs
  investigation.
- **45 rows** matched more than one plausible source measurement with genuinely different CI
  bounds across candidates (ambiguous which one is correct) — not auto-backfilled, left as 0/0
  pending manual disambiguation.

Method: for each of the 18 identifiable trials, fetched the current
`?fields=resultsSection` response, indexed every `GEOMETRIC_MEAN` measurement by
(outcome-measure title → assay, class title → serotype, groupId → vaccine/arm), and matched
each 0/0 CSV row to a source measurement by serotype + assay + point-value agreement (allowing
for the class-title formatting quirks seen in the source, e.g. missing spaces like
"Serotype19A"). A match was only auto-backfilled when either exactly one candidate was found,
or multiple candidates were found but all agreed on the same CI bounds.

One incidental discovery from this pass: NCT04665050 (Merck V116/PCV21) reports the literal
string `"NA"` as the `lowerLimit`/`upperLimit` value for 54 measurements in its API
`resultsSection` (the key is present, but its value is the text "NA", not a number) — these
were correctly carried through as `NA` in the CSV rather than misread as "has a numeric CI."

## Update — 2026-07-20 (continued): log-normal CI derivation for NCT02736240

Per user request: NCT02736240's manuscript (Zhao et al., *Front Microbiol.* 2022;13:870973,
[doi:10.3389/fmicb.2022.870973](https://doi.org/10.3389/fmicb.2022.870973)) reports no
individual-group CI for GMC or OPA GMT (see above), but it does report, at the same N, the % of
subjects reaching the WHO/assay seroprotection threshold (IgG ≥0.35 μg/mL in Table 3; OPA ≥1:8 in
Table 5). Assuming individual titers are log-normally distributed, that percentage plus the GMC/GMT
is enough to back out the underlying SD of log-titer, and from that a 95% CI for the group mean:

- σ(ln) = (ln(threshold) − ln(GMC)) / Φ⁻¹(1−p), where p = proportion ≥ threshold
- SE = σ(ln) / √N
- 95% CI = exp(ln(GMC) ± 1.96·SE)

This is a **derived/estimated** CI, not a value transcribed from the source — logged as such,
distinct from a directly-reported CI.

Applied to all 104 rows (both assays, both timepoints present in the CSV: "1m post primary child"
and "1m post boost child"). Of these, **63 were derived**; **41 were left at `NA`** because the
method breaks down in three ways:
- **36 rows**: % responders = 100% or 0% exactly → Φ⁻¹ is infinite, no finite SD exists.
- **4 rows**: reported %<50% while GMC is above threshold (or vice versa) → the formula returns a
  negative SD, i.e. the manuscript's own % and GMC are in tension under the log-normal assumption
  for that specific cell. Left `NA` rather than reporting a physically meaningless bound.
- **1 row** (OPA, serotype 7F, PCV7, toddler dose): % responders (52.58%) is close enough to 50%
  that Φ⁻¹(1−p) is near zero, blowing the derived SD up to an implausible value (a 95% CI spanning
  4 orders of magnitude, vs. a maximum ratio of ~5× across all 63 legitimately-derived rows).
  Rejected via a sanity cap (upper/lower ratio > 20×) rather than reported.

Full row-by-row log (assay, serotype, vaccine, timepoint, GMC/GMT, % responders, N, derived σ/SE,
derived bounds, and status/reason): `data/ci_lognormal_derived_2026_07_20.csv`.

**Incidental finding while cross-checking Table 5 for this pass**: 10 of the 13 serotypes' OPA
rows for PCV7 at "after infant series" had the *PCV13* GMT value duplicated into the PCV7 row
instead of PCV7's actual value from Table 5 (only serotypes 4, 9V, and 6B had correct, distinct
PCV7 values). This is a point-value bug, independent of the missing-CI issue, and was corrected
from Table 5 for those 10 rows.

Also fixed while in this block: `outcome_overview_time_frame_weeks` was `0` for all 104 rows
regardless of timepoint ("1m post primary child" and "1m post boost child" both), violating the
project's own convention that "1m post…" timepoints map to `4` weeks, not `0`. Corrected to `4`
for all 104 rows. Participant counts (N) were also backfilled from Table 3/4/5 for all 104 rows
(previously blank).

## Update — 2026-07-20 (continued): the remaining 61 rows (45 "ambiguous" + 16 "unmatched") resolved

Root cause of the "45 ambiguous" classification: the matching pass above matched candidate source
measurements by **serotype + assay + point-value agreement only**, without using the group
identity already present in every CSV row (`outcome_overview_id` / `outcome_overview_title`).
Group ID codes (`OG000`, `OG001`, ...) are re-scoped per outcome measure — the same code means a
different arm in a different table within the same trial — so when two distinct treatment groups
happened to round to the same point value but had different CI bounds, the value-only match looked
unresolvably ambiguous. Re-matching by **source group title** (exact string match against
`outcome_overview_title`) + serotype + unit-type-consistent assay (titer units for OPA,
concentration units for GMC), with `outcome_overview_dose_number` as a tiebreaker where a trial
reports the same group across multiple dose timepoints, resolved all 61 rows to a single
unambiguous source measurement — no genuine ambiguity remained once the already-captured group
identity was used.

This pass also caught value-level bugs beyond CI bounds:

- **NCT00373958**: all 6 rows in this bucket (not just the 1 flagged in the original audit) had
  `outcome_overview_value = 0`, not matching source at all. All 6 corrected (value + bounds) from
  the "After the 3-Dose Infant Series and the Toddler Dose" OPA GMT table, group
  "13vPnC After the Infant Series".
- **NCT04384107**: serotype 33F OPA value was `0`; corrected to `19899.34` (95% CI 14204.44–27877.46)
  from the "GMT of Serotype-specific OPA... After Dose 4" table.
- **NCT00366899**: 9 rows were missing not only CI bounds but `outcome_overview_assay`,
  `outcome_overview_manufacturer`, `outcome_overview_dose_description`, and
  `outcome_overview_schedule` as well (all blank) — this is why the original pass's assay-filtered
  matching never found them. Backfilled from sibling rows in the same "After Toddler Dose" GMC
  table (`assay=GMC`, `manufacturer=Pfizer`, `dose_description=1m post boost child`,
